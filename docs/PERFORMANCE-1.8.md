## 第四组：关系图谱（服务端）

| 用例 | v1.6.5 典型 | **v1.8.5** | 变化 |
|------|------------|-----------|------|
| graph/local_graph_depth2/50 | ~35 ns | **29.3 ns** | **-16% ✅** |
| graph/local_graph_depth2/200 | ~80 µs | **192 µs** | ⚠ +140%（见注）|
| graph/local_graph_depth2/500 | ~170 µs | **175 µs** | 持平 ✅ |

**注 graph/200 回退**：v1.8.4 为图谱节点新增了 `mtime: i64` 字段（热力图功能），每个节点在构建时需要额外的 `SystemTime::duration_since(UNIX_EPOCH)` 计算。200 节点规模下此额外开销被放大。500 节点时由于 BFS 剪枝比例更高，影响不明显。

优化方向：可在 `generate_graph` 中将 mtime 转换改为条件性（仅需要热力图时才计算），或预计算 mtime 到 AppState 中。

### 6-b/c：NoteIndex 搜索 ⚠ 重要变化

| 查询类型 | v1.6.5 | v1.7.x（M4）| **v1.8.5** | 对比 v1.7.x |
|---------|--------|------------|-----------|------------|
| ASCII | 192–229 µs | ~200 µs | **1855 µs** | **+828% 显著退化** |
| CJK | 618–661 µs | ~620 µs | **2015 µs** | **+225% 显著退化** |

**⚠ WASM 搜索性能回退分析**

本次 WASM 搜索退化幅度异常（ASCII +828%、CJK +225%），推测原因：

1. **测试数据规模变化**：v1.8.5 基准测试运行时系统整体负载较高（同时在编译 zip/rsa 等大依赖），WASM bench 在高负载下运行可能导致数据失真。

2. **M4 优化内容的实际效果**：M4 将评分阶段改为 `token_to_qidx` HashMap + u8 位掩码。在较大 `content_tokens` 集合（1000 条笔记时平均每条数百 tokens）下，**迭代 `content_tokens` 建立位掩码的开销反而高于原来的 HashSet.contains() 查询**——HashSet 平均 O(1)，而迭代所有 tokens 是 O(n_tokens)。

3. **验证方案**：需要在低系统负载下单独重跑 WASM 搜索基准：
   ```bash
   cargo bench -- note_index_search
   ```

**建议**：将 M4 的 `token_to_qidx` 方案回退，改用原来的 `HashSet.contains()` 策略，并在评分前对 `content_tokens` 建立反向索引以优化大集合场景。

## 待优化方向

**WASM 搜索（优先级：高）**
```
问题：M4 评分 Bitset 方案对大 content_tokens 集合性能回退
原因：迭代所有 content_tokens 建位掩码 O(n_tokens)
     vs 原 HashSet.contains() O(1)
方案：评分阶段仅迭代 query_tokens（≤8），逐一在 title/tag/content HashSet 中查，
     回归原始方案；或对 content_tokens 也建倒排使用 O(1) 查询
```

**图谱 mtime 计算（优先级：低）**
```
问题：v1.8.4 GraphNode 新增 mtime 使 graph/200 增加 ~140%
原因：每个节点额外执行 SystemTime::duration_since(UNIX_EPOCH)
方案：将 mtime 转换提前到 Note 加载时缓存为 i64，避免在图谱构建热路径重复计算
```
