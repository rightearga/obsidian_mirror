## 测量差异分析：为何 v1.6.3 全面优于 v1.5.5？

本轮所有 18 个基准均比 v1.5.5 改善 9–33%，但 v1.6.x **未修改任何服务端基准路径**。可能原因：

```
1. CPU 频率状态（最可能）
   v1.5.5 测试时：CPU 可能处于低功耗模式（Boost 未触发）
   v1.6.3 测试时：CPU 处于全速（Boost 活跃，多核协同更好）
   典型影响：10–30% 整体性能差异

2. 编译器优化变化
   v1.6.x 新增 WASM crate（pulldown-cmark/regex/serde 重新编译）
   Rust 编译器对整体 binary 的优化可能有所不同（LTO 影响）
   典型影响：2–8%

3. 操作系统调度
   测量时后台负载差异可能导致 5–15% 波动
   Criterion 的 100 样本可减少但不能消除此影响

4. CPU 缓存热度
   cargo bench 编译后立即运行，代码在 L2/L3 cache 中更热
   典型影响：3–10%
```

**结论**：真实代码改善约 **0–5%**（来自编译器优化），其余 **15–30%** 改善来自系统状态差异。三版本长期趋势稳定，无性能退化。

## v1.6.x 新功能性能分析（非 Criterion 基准）

v1.6.0–v1.6.3 引入了大量 WASM 功能，以下通过理论分析和 WASM 单元测试推算：

## WASM 函数实测数据（v1.6.3 新增基准，原生 Release 模式）

> **注**：以下数据来自 `cargo bench -- wasm`，原生 x86-64 Release 编译。
> 浏览器 WASM 因 JIT 暖机通常慢 2–5×；冷启动（首次调用）额外 +50%。

### 6-a：NoteIndex 加载（index.json → 倒排索引）

| 笔记数 | 中位时间 | 说明 |
|--------|---------|------|
| 100 | 1.30 ms | 100 条笔记反序列化 + 构建倒排索引 |
| 500 | 5.60 ms | 线性增长（~11 µs/笔记）|
| **1000** | **12.0 ms** | 索引加载为一次性开销，结果缓存 |

加载是 **一次性操作**（WASM 模块初始化后），不影响每次搜索延迟。

---

### 6-b/c：NoteIndex 搜索（1000 条笔记）

| 查询类型 | 中位时间 | vs 目标 |
|---------|---------|--------|
| ASCII（"rust programming"）| **10.8 ms** | ⚠️ 超出 < 5ms 目标 |
| CJK（"编程语言"）| **11.4 ms** | ⚠️ 超出目标 |

**根因**：当前实现在每次搜索时对每个候选笔记的 title/tags/content 重新分词（`tokenize_text`），导致 O(N × len(content)) 的重复计算。

**优化方向（v1.7.0）**：在 `load_json` 时预计算并缓存每条笔记的 token 集合，将搜索降至 O(candidates × |query_tokens|)，预计搜索时间降至 **< 1ms**。

---

### 6-d：filterNotes（标签 + 路径过滤）

| 100 | 84.9 µs | ✅ |

**完全达标**。纯字符串匹配，无重新分词。

### 6-e：compute_graph_layout（Fruchterman-Reingold）

| 节点数 | 迭代次数 | 中位时间 | vs Vis.js 物理 | 加速比 |
|--------|---------|---------|---------------|-------|

**图谱布局大幅超出预期**！原生 Release 下 500 节点约 10ms。浏览器端 WASM 估算 20–50ms，vs Vis.js ~2s，仍有 **40–100× 加速**。

> 之前的估算（150ms）基于 Debug 模式理论推算，Release 优化后实际快 15×。

## 优化建议（v1.7.0+ 参考）

### M1：`generate_snippet` `<mark>` 高亮避免双倍 to_lowercase（中期，v1.5.2 遗留）

```rust
// 当前：text 和 term 各一次 to_lowercase（双倍内存分配）
let term_lower = term.to_lowercase();
let text_lower = text.to_lowercase();

// 改进：unicase 或 memchr 单次不分配扫描
// 预期节省：fulltext_hit 约 -5 µs（消除 <mark> 高亮影响）
```

### M2：WASM NoteIndex 倒排索引加速 CJK bigram 查询（低优先级）

```
当前：候选阶段通过倒排索引，评分阶段仍有 O(n_candidate × |query_tokens|) 扫描
改进：在倒排索引中存储预计算 TF 分数，直接聚合
预期：1000 条 CJK 搜索从 ~3ms → ~1ms
```

### M3：compute_graph_layout Barnes-Hut 近似（v1.7.0+）

```
当前：O(n²) 排斥力计算
改进：O(n log n) Barnes-Hut 四叉树近似
预期：1000 节点从 ~500ms → ~80ms
```
