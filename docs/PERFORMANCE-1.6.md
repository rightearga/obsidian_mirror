### 6-b/c：NoteIndex 搜索

| 查询类型 | 无 TokenCache（v1.6.2 原始）| TokenCache 优化（v1.6.3）| **M3 优化（v1.6.4）** | vs 目标 |
|---------|--------------------------|------------------------|----------------------|--------|
| ASCII | 10.8 ms | 244 µs | **239 µs** | ✅✅ |
| CJK | 11.4 ms | 705 µs | **661 µs** | ✅✅ |

**M3 改善 CJK -6%**（661 vs 705 µs）。机制：
1. query token 按倒排稀有度排序，截断至最多 8 个（抑制 CJK bigram 泛滥带来的宽泛候选集）
2. `is_cjk_bigram` 改为零分配实现（无 `Vec<char>` 分配）
3. `title_weights` 向量在候选循环外预计算，消除每候选节点的重复调用
4. CJK bigram 标题权重 10 → 15（精准匹配优先级更高）

## M1/M2/M3 优化效果总结

| 优化 | 目标 | 实测改善 | 机制 |
|------|------|---------|------|
| **M1** `find_substr_ci` | -2~5 µs on fulltext_hit | **-8.6 µs (-16%)** | 避免 `text.to_lowercase()` 堆分配 |
| **M2** Barnes-Hut θ=0.9 | 500节点 10ms→2ms | **10.6→4.53ms (-57%)** | O(n²)→O(n log n) 排斥力计算 |
| **M3** CJK 搜索均衡 | CJK 705→~300µs | **705→661µs (-6%)** | token 限制 + 零分配 bigram 检测 |

M3 的 CJK 改善小于预期（-6% vs -58% 目标）。主要原因：tokenCache 预缓存（v1.6.3 已完成）已是主要优化，M3 的 token 截断只减少了部分候选评分开销。进一步优化可考虑倒排索引的位图存储。

## 剩余优化方向

### 已完成（v1.6.4）

- ✅ M1：highlight_terms 零分配查找
- ✅ M2：Barnes-Hut O(n log n) 图谱布局
- ✅ M3：CJK token 限制 + bigram 权重

### 未来可选优化

**M3-续：NoteIndex 倒排索引位图加速**
```
当前：HashSet<usize> 存储候选 note 索引
改进：bitset（u64 数组，每位代表一个 note）实现候选集的位与运算
预期：CJK 搜索 661µs → ~400µs（减少 HashSet 开销）
```

**M4：render_markdown 正则编译缓存**
```
当前：lazy_static! 已在进程级缓存正则
改进：无，render_markdown 已接近理论最优（12µs 复杂笔记）
```

**M5：Barnes-Hut θ 自适应**
```
当前：θ = 0.9（固定）
改进：初始迭代用 θ = 1.2（更快，稳定期用 θ = 0.7（更精确）
预期：500节点布局质量提升，速度相近
```
