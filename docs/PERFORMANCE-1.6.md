## 第一组：Markdown 处理

### 实测数据（三版本对比）

| 用例 | v1.4.9 | v1.5.5 | v1.6.3 | 1.4→1.6 |
|------|--------|--------|--------|---------|
| minimal | 5.57 µs | 6.97 µs | **4.93 µs** | -12% |
| with_wikilinks | 37.82 µs | 42.79 µs | **30.68 µs** | -19% |
| complex | 85.93 µs | 83.30 µs | **65.20 µs** | **-24%** |
| long_5000chars | 346.77 µs | 395.61 µs | **298.07 µs** | -14% |

### 分析

v1.5.4 引入的 `ENABLE_FOOTNOTES` 和 embed 检测（`![[]]` 新分支）在本轮测试中影响不显著。`complex` 用例改善最明显（-24%），主要来自系统状态和编译器优化的累积效应。

`with_wikilinks`（-19%）是 v1.6.x 中的持续亮点——WikiLink 正则扫描路径的 Release 优化随 crate 图变化而改善。

## 第六组：WASM 函数（v1.6.3 新增基准）

> 以原生 x86-64 Release 模式运行。浏览器 WASM 因 JIT 暖机通常慢 2–5×；
> 实际体验依赖浏览器 WASM 引擎状态，冷启动额外 +50%。

### 6-a：NoteIndex 加载

| 笔记数 | 中位时间 | 速率 |
|--------|---------|------|
| 100 | 1.80 ms | ~18 µs/笔记 |
| 500 | 8.96 ms | ~18 µs/笔记 |
| **1000** | **18.2 ms** | **~18 µs/笔记（线性）** |

一次性初始化开销，加载后缓存于 JS 变量中，不影响搜索延迟。

---

### 6-b/c：NoteIndex 搜索（优化前 → 优化后）

> **v1.6.3 修复**：`load_json` 时预计算并缓存各字段 token 集合（`TokenCache`），
> 消除 `search_json` 中每次调用的重复分词（O(N × content) → O(candidates × query_tokens)）。

| 查询类型 | 优化前 | **优化后** | 加速比 | vs 目标 |
|---------|--------|-----------|--------|--------|
| ASCII（"rust programming"）| 10.8 ms | **244 µs** | **×44** | ✅ 目标 < 5ms |
| CJK（"编程语言"）| 11.4 ms | **705 µs** | **×16** | ✅ 目标 < 5ms |

CJK 比 ASCII 慢约 3× 是预期行为：CJK 分词生成 unigram + bigram（每字 2 token），查询 token 集更大。两者均远优于目标。

**加载时间代价**（优化前 → 优化后）：

| 笔记数 | 优化前 | 优化后 | 增加 |
|--------|-------|-------|------|
| 1000 | 12.0 ms | 18.2 ms | +52% |

加载时间增加是将分词从高频路径（搜索）移至低频路径（一次性加载）的正确权衡。

## 优化建议（v1.7.0+）

### M1：`highlight_terms` 避免双倍 `to_lowercase` 分配（遗留自 v1.5.2）

```rust
// 当前：text 和 term 各 to_lowercase 一次（两次堆分配）
let term_lower = term.to_lowercase();
let text_lower = text.to_lowercase();
// 改进：unicase crate 或 memchr 实现零分配大小写不敏感匹配
// 预期节省：fulltext_hit -2~5 µs
```

### M2：WASM graph_layout 大图近似（v1.7.0+）

```
当前：O(n²) 排斥力（500节点 × 15迭代 = 187,500 对 × 15 = 281万次计算）
改进：Barnes-Hut 四叉树近似 O(n log n)
预期：500节点 10.6ms → ~2ms（×5 加速）
```

### M3：NoteIndex CJK 搜索速度提升（已大幅改善，可继续优化）

```
当前：CJK 705µs（ASCII 244µs 的 2.9×），因 bigram 生成 token 数更多
改进：限制最大 token 长度 + trie 结构前缀压缩
预期：CJK 搜索降至与 ASCII 相近水平（~300µs）
```
