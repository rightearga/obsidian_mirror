### 6-b/c：NoteIndex 搜索（Bitset 加速，v1.6.5）

| 查询类型 | v1.6.3（TokenCache 前）| v1.6.3（TokenCache 优化）| v1.6.4（M3 优化）| **v1.6.5（Bitset）** | 累计改善 |
|---------|---------------------|----------------------|----------------|---------------------|---------|
| ASCII | 10.8 ms | 244 µs | 239 µs | **192–229 µs** | **×47–56** |
| CJK | 11.4 ms | 705 µs | 661 µs | **618–661 µs** | **×17–18** |

> 数值范围来自多次运行（最优 192µs / 典型 228µs for ASCII；最优 618µs / 典型 661µs for CJK）。

**Bitset 改善机制**：
```
旧方案（HashSet）：
  n 次 HashSet.insert(idx)
  = n 次哈希计算 + 随机内存访问（cache miss 概率高）

新方案（Bitset）：
  bits[idx/64] |= 1u64 << (idx%64)
  = 1 次位运算（无哈希，内存访问连续，16个u64=128字节全 L1 缓存）
```

**ASCII 改善明显（-4~-19%）**：ASCII 查询词（如 "rust"）命中的 posting list 较大（常见词），HashSet 需要大量随机插入；Bitset 的位或运算完全消除这部分开销。

**CJK 改善较小（-0~-6%）**：CJK bigram 是稀有词，posting list 很小，HashSet 插入次数本来就少；主要瓶颈仍在后续的候选评分阶段，Bitset 只消除了较小部分开销。

## 剩余优化方向

**NoteIndex CJK 搜索进一步均衡**
```
当前：CJK ~620µs（ASCII ~200µs，差距 3×）
瓶颈：评分阶段对每个候选笔记遍历所有 query_tokens 进行 HashSet 查询
改进：scoring 阶段也改用位运算加速，或对 query 做更激进的 token 过滤
预期：CJK ~400µs（接近 ASCII 水平）
```

**Barnes-Hut θ 自适应**
```
当前：θ = 0.9（固定）
改进：前期迭代 θ = 1.2（快速收敛），稳定期 θ = 0.7（精细布局）
预期：500节点 ~5ms → ~3ms，布局质量更好
```
